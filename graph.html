Wine Graph

Simple editable diagrams

This is intentionally not a strict ERD. It is a lightweight product-and-data view that is easier to edit in HTML, easier to restyle for branding, and easier to hand to Codex inside WebStorm.

Core wine data shape

A softer, more presentation-friendly view of how producer, area, wine, and wine references relate.
Canonical ownership stays with producer
Area can represent country, region, AVA, AOC, subregion
Wine refs point to external systems or imports

Producer

The source-of-truth owner for canonical wine identity.

  • name
  • slug
  • producer type
  • home area
  • story / profile

Area

A simple hierarchy for place and appellation context.

  • name
  • type
  • parent area
  • country / region / subregion
  • display path

Wine

The canonical wine record used across the graph.

  • name
  • vintage
  • producer
  • primary area
  • color / type / closure / bottle data

Wine Ref

Links the canonical wine to outside systems or ingested source data.

  • source name
  • source id
  • sku or handle
  • match status
  • import metadata
Producer can be based in an area
Area can nest under another area
Producer creates many wines
Wine belongs to one primary area
Wine can have many refs

Plain-English relationship view

Producer → Wine A producer owns and publishes one or more canonical wines.
Area → Producer / Wine An area gives geographic context. A producer may have a home area, and a wine may have an origin area.
Wine → Wine Ref A wine can be linked to multiple retailer, POS, import, or adapter records without losing canonical identity.

Good optional add-ons later

Grape Separate many-to-many relationship if you want varietal composition shown explicitly.
Vintage details Harvest, bottling, release, and aging data if you want a deeper producer-side model.
Retailer listing Keep retailer inventory separate from canonical wine and link through refs or matches.
Editing tip: duplicate cards, rename headings, and move blocks around with grid spans first. Codex can then add arrows, SVG connectors, icons, brand color tokens, or a much more polished layout later.

Wine Graph user types and functionality

This is more product-facing than technical. It shows who the main user types are and what each one does in the system.

Visitor / Consumer

Lightweight discovery user.

  • search wines and producers
  • browse regions and areas
  • view retailer availability
  • save or follow later

Retailer User

Inventory and discovery operator.

  • connect POS or commerce source
  • sync inventory
  • review matches
  • improve discoverability

Producer User

Canonical data owner and publisher.

  • create and manage wines
  • maintain producer profile
  • publish accurate wine data
  • track downstream presence

Admin / Operator

Internal stewardship and quality control.

  • moderate data issues
  • review bad matches
  • monitor adapters and sync jobs
  • support onboarding

Core functionality by role

  • Visitor / Consumer: discovery, search, nearby retail availability, producer browsing
  • Retailer: source connection, inventory sync, match review, storefront visibility
  • Producer: canonical data publishing, wine updates, brand presence
  • Admin: data governance, support, adapter operations, merge review

Simple platform framing

  • Producers improve the graph by publishing accurate canonical wine data.
  • Retailers improve the graph by linking real inventory to that canonical data.
  • Consumers benefit from cleaner discovery and better availability signals.
  • Operators keep quality high when data is messy or ambiguous.
